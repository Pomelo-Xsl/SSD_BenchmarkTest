# NVMe SSD Benchmark MVP 操作说明

本文说明如何在 Linux 测试机上启动并验证 NVMe SSD Benchmark MVP。请仅使用专用测试机和测试盘。

> **重要警告：** 写入类 fio 测试会覆盖目标 NVMe SSD 上的数据。不要对系统盘、已挂载磁盘或任何含有重要数据的磁盘执行测试。

## 1. 前置条件

- Linux 系统（建议 Ubuntu/Debian）。
- Python 3.11 或更新版本。
- 用于测试的独立 NVMe SSD，且设备及其分区均未挂载。
- 启动实际测试服务时具有 `root` 权限。

## 2. 安装依赖

进入项目目录并创建 Python 虚拟环境：

```bash
cd "/Users/pomelo/Documents/SSD_BenchmarkTest"

python3.11 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

安装系统工具（Ubuntu/Debian 示例）：

```bash
sudo apt update
sudo apt install -y fio nvme-cli util-linux
```

确认工具可用：

```bash
fio --version
nvme version
lsblk --version
```

## 3. 启动服务

实际执行 fio 前，服务必须以 root 身份运行；否则任务会被安全检查拒绝。

```bash
cd "/Users/pomelo/Documents/SSD_BenchmarkTest"
sudo "$(pwd)/.venv/bin/uvicorn" app.main:app --host 127.0.0.1 --port 8000
```

服务启动后可访问交互式 API 文档：

```text
http://127.0.0.1:8000/docs
```

## 4. 验证服务健康状态

另开一个终端，执行：

```bash
curl http://127.0.0.1:8000/health
```

预期响应：

```json
{"status":"ok"}
```

## 5. 扫描 NVMe 设备

```bash
curl http://127.0.0.1:8000/api/devices
```

响应中会列出 NVMe 设备名称、路径、型号、序列号、容量、固件和温度等信息。例如：

```json
[
  {
    "name": "nvme1n1",
    "path": "/dev/nvme1n1",
    "model": "Example SSD",
    "serial": "ABC123",
    "size_bytes": 1000204886016,
    "firmware": "1.0",
    "temperature_c": 35.0
  }
]
```

扫描结果还包含安全状态：`safe_to_test` 为 `true` 才可测试；`mounted` 表示设备或分区已挂载，`system_disk` 表示系统盘，`has_partitions` 表示已有分区。`safety_message` 会给出不可测试的原因。

在创建任务前，务必人工确认目标不是系统盘，并检查它没有挂载分区：

```bash
lsblk -o NAME,PATH,MOUNTPOINTS
findmnt /
```

## 6. 运行只读测试（推荐先执行）

将下面的 `nvme1n1` 替换为扫描结果中的设备名。以下测试不会写入磁盘：

```bash
curl -X POST http://127.0.0.1:8000/api/tests \
  -H 'content-type: application/json' \
  -d '{"test_name":"rand_read_4k"}'
```

系统已在项目根目录的 `.env` 中配置默认测试盘 `nvme1n1`。如需临时改用其他盘，也可在请求中显式传入 `device_name`。

可选的只读测试类型：

- `seq_read_128k`：128 KiB 顺序读取。
- `rand_read_4k`：4 KiB 随机读取。

### 调整 fio 参数

可在请求的 `fio_options` 中按任务覆盖默认 fio 参数。例如，将测试设为 120 秒、预热 15 秒、队列深度 64：

```json
{
  "device_name": "nvme1n1",
  "test_name": "rand_read_4k",
  "fio_options": {
    "runtime_seconds": 120,
    "ramp_time_seconds": 15,
    "iodepth": 64,
    "numjobs": 1,
    "ioengine": "io_uring",
    "direct": true
  }
}
```

Swagger 会提示以下常用字段：

- `runtime_seconds`：测试时长，1–86400 秒。
- `ramp_time_seconds`：预热时长，0–3600 秒。
- `iodepth`：队列深度，1–1024。
- `numjobs`：并发 fio 任务数，1–128。
- `ioengine`：任意 fio 支持的 I/O 引擎名称。
- `direct`：是否绕过系统页缓存。

未填写的字段使用服务默认值。除此以外，可直接在 `fio_options` 中添加任意 fio 参数，例如：

```json
"fio_options": {
  "bs": "16k",
  "rw": "randread",
  "size": "100G",
  "norandommap": true,
  "randrepeat": false
}
```

系统会把它们转换为 fio 命令行参数。出于安全和结果解析需要，不能覆盖 `filename`、`name`、`output`、`output-format`、`time_based`，也不允许设置执行外部命令的 fio 参数。若将 `rw` 改成除 `read` 或 `randread` 外的模式，必须将 `confirm_destructive` 设为 `true`。

成功创建后，接口会返回任务。例如：

```json
{"id":1,"status":"queued"}
```

## 7. 查询测试结果

使用同一个接口查询测试状态和完成后的指标：

```bash
curl http://127.0.0.1:8000/api/results/1
```

任务状态依次为 `queued`、`running`、`completed`；失败时为 `failed`，并通过 `error_message` 返回原因。完成时，`result` 字段包含：

- `iops`：每秒 I/O 次数。
- `bw_mib_s`：带宽（MiB/s）。
- `latency_avg_us`：平均延迟（微秒）。
- `latency_p99_us`：P99 延迟（微秒）。
- `cpu_user_pct`、`cpu_system_pct`：fio 进程 CPU 使用率。

## 8. 写入测试（危险）

仅可在已确认为空的专用测试盘上执行。写入任务必须显式设置 `confirm_destructive` 为 `true`：

```bash
curl -X POST http://127.0.0.1:8000/api/tests \
  -H 'content-type: application/json' \
  -d '{"device_name":"nvme1n1","test_name":"rand_write_4k","confirm_destructive":true}'
```

可选的写入测试类型：

- `seq_write_128k`：128 KiB 顺序写入。
- `rand_write_4k`：4 KiB 随机写入。

服务会在运行 fio 前检查 root 权限、设备存在性、NVMe 设备类型、挂载状态、系统盘风险以及 fio/nvme-cli 是否已安装。任一检查失败，任务将标记为 `failed`。

## 9. 查看文件与日志

- SQLite 数据库：`benchmark.db`。
- fio 原始 JSON：`results/task_<任务ID>.json`。
- 应用日志：`logs/app.log`。
- fio 运行日志：`logs/fio.log`。
- 错误日志：`logs/error.log`。

遇到问题时，优先查看任务接口的 `error_message` 和 `logs/error.log`。
